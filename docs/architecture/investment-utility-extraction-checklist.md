# Investment Utility Extraction Checklist

## 1. Title and purpose

This document is the **Phase 5E pure utility extraction checklist** for
`js/investment-analytics.js`. It is a documentation-only checklist to use before the first
runtime PR that extracts pure helpers from the current Investment Intelligence analytics
entrypoint.

Phase 5E does **not** move, refactor, rename, relink, or create runtime JavaScript or CSS. It
records the current candidate utility inventory, loading compatibility constraints, first safe
runtime scope, and browser smoke tests that should be used by a later extraction PR.

## 2. Why utilities first

Pure utilities should be extracted before larger runtime features because they have the smallest
surface area and the clearest behavior contract:

- Pure helpers are lower risk than charts, calculators, chatbot behavior, premium gate behavior,
  or sector selector/panel logic.
- Utilities are easier to test and review because inputs and outputs can be compared without a
  browser DOM, Firebase session, network request, or visual baseline.
- Extracting utilities gradually turns `js/investment-analytics.js` into a thinner orchestrator
  while preserving the current public script entrypoint and page behavior.

## 3. Current source file to inspect

Phase 5E inspected these files without editing runtime pages, JavaScript, CSS, data, scripts,
workflows, route files, or package configuration:

- `js/investment-analytics.js`
- `pages/investment-analytics.html`
- `pages/investment-retail.html`
- `pages/investment-airlines.html`
- `pages/financial-indicators.html`
- Existing Phase 5C/5D architecture docs and analytics script utility README skeletons.

Investment-related CSS was not needed for this documentation update beyond the Phase 5C rendering
and class-dependency notes because Phase 5E does not change rendering or stylesheet ownership.

## 5. Strict definition of pure utility

For Phase 5E and the next runtime extraction PR, a pure utility is a function that:

- Does not read or write DOM.
- Does not add or remove event listeners.
- Does not call `fetch`.
- Does not read or write `localStorage` or `sessionStorage`.
- Does not depend on Firebase, auth state, `window`, or `document`.
- Does not mutate global or shared state.
- Does not render HTML or SVG into the page.
- Returns the same output for the same input.
- Can be moved without changing execution order.

## 6. Functions that must not be extracted first

The first runtime extraction PR must avoid these categories:

- Chart renderers.
- SVG element creation if tightly coupled to DOM rendering.
- Calculator render/update functions.
- Event handlers.
- Boot/init functions.
- Data fetchers.
- Chatbot UI behavior.
- Premium gate behavior.
- Sector selector/panel logic.
- Functions depending on global constants or mutable state unless those dependencies are moved
  safely too.

## 7. Proposed first runtime extraction scope

The next runtime PR after this checklist should be deliberately tiny:

- Create `domains/investment-intelligence/analytics/scripts/utils/formatters.js`.
- Move only 1-3 clearly pure formatter or number helper functions if discovered. The best first
  candidates are `formatNumber`, `formatUsdCompact`, `formatTlMillion`, `formatUsdCurrency`, or
  `parseLocalizedNumber`.
- Keep `js/investment-analytics.js` as the public entrypoint.
- Import from the new module only if script loading compatibility allows it.
- If direct import is unsafe because `js/investment-analytics.js` remains a classic script, create
  a legacy-safe wrapper strategy first.
- Do not extract chart, calculator orchestration, sector panel, chatbot, premium gate, fetcher, or
  renderer logic in the first runtime PR.

Direct ES imports are **not safe** inside the current `js/investment-analytics.js` file while it
is loaded as a classic deferred script. A later runtime PR should either:

1. Add a compatibility/loading plan to convert the public entrypoint to `type="module"` and verify

2. Keep the classic entrypoint and add a legacy-safe bridge/wrapper that exposes moved helpers on a
   stable namespace before the classic script consumes them.

No such bridge, module conversion, or script tag change should be part of Phase 5E.

## 11. Risk notes

- Utility functions may close over global constants.
- Formatting changes can visually alter charts and tables.
- Number parsing changes can alter calculator outputs.
